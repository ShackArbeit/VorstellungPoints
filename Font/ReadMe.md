# 跨境電商公司工程師面試最常問的 12 個技術題

> 適用職缺情境：  
> 全端工程師 / Node.js 工程師 / 跨境電商公司內部系統工程師  
> 面試公司類型：自有官網 + Amazon / 跨境平台營運 + 後台系統維護 + 資料流程整合

---

## 1. 如果公司有自家官網、Amazon 平台、內部管理後台，資料要怎麼設計才不會混亂？

### 面試官想知道什麼
- 你是否理解電商系統不是只有前台頁面
- 你是否知道多平台資料同步的困難點
- 你有沒有資料模型設計觀念

### 建議回答
如果公司同時經營自家官網、Amazon 平台與內部後台，我會先把資料模型的核心拆清楚，避免不同平台各自維護一份資料，最後導致資料不同步。

我通常會把資料分成幾個層次：

1. **商品主檔（SPU）**
   - 商品名稱
   - 品牌
   - 規格描述
   - 類別
   - 共用圖片或共用屬性

2. **可販售單位（SKU）**
   - SKU code
   - 售價
   - 重量
   - 尺寸
   - 庫存
   - 條碼
   - 各平台對應 ID

3. **平台對應表**
   - Amazon listing id
   - 官網商品 id
   - 上下架狀態
   - 平台價格
   - 平台標題
   - 平台專屬描述

4. **訂單資料**
   - 訂單主表
   - 訂單明細
   - 付款狀態
   - 出貨狀態
   - 平台來源

5. **庫存異動紀錄**
   - 入庫
   - 出貨
   - 退貨
   - 調整原因
   - 操作人員

我的原則會是：
- **主資料集中管理**
- **平台差異拆表處理**
- **所有同步都保留紀錄**
- **避免手動修改多份資料**

這樣做的好處是，未來不管新增 Walmart、eBay 或其他平台，都可以沿用既有架構擴充。

---

## 2. 如果商品資料要從 Excel 匯入系統，你會怎麼設計？

### 面試官想知道什麼
- 你有沒有實作匯入功能的經驗
- 你是否理解資料驗證的重要性
- 你會不會只做「能上傳」而不考慮資料品質

### 建議回答
我會把 Excel 匯入功能分成幾個明確階段，而不是讓使用者直接上傳後寫進資料庫。

### 我通常會這樣設計：

#### 1. 範本定義
先提供固定欄位格式，例如：
- 商品名稱
- SKU
- 價格
- 庫存
- 重量
- 類別
- 圖片 URL

這樣可以降低使用者亂填的機率。

#### 2. 上傳後先做預檢
不直接寫入正式資料表，而是先做：
- 必填欄位檢查
- 型別檢查
- 價格是否為數字
- SKU 是否重複
- 類別是否存在
- 圖片格式是否合法

#### 3. 顯示錯誤報告
如果資料有問題，要讓使用者知道：
- 第幾列錯誤
- 哪個欄位錯誤
- 錯誤原因是什麼

#### 4. 預覽後再正式匯入
讓使用者確認後再匯入，避免錯誤資料直接污染正式環境。

#### 5. 保留匯入紀錄
包括：
- 匯入人
- 匯入時間
- 成功筆數
- 失敗筆數
- 原始檔案
- 錯誤明細

### 延伸加分回答
如果資料量很大，我會改成：
- 上傳後進入背景 job
- 非同步處理
- 匯入完成後寄通知或在後台顯示結果

這樣不會讓前端 request timeout，也比較符合實務使用情境。

## 3. 如何設計一個可維護的 Node.js / RESTful API？

### 面試官想知道什麼
- 你有沒有後端分層概念
- 你是否知道什麼叫「可維護」
- 你會不會把所有邏輯都寫在 controller

### 建議回答
我會把 API 設計分成幾個層次處理，避免所有邏輯都混在一起。

### 常見分層方式
- **Controller**
  - 接收 request
  - 驗證輸入
  - 回傳 response

- **Service**
  - 處理商業邏輯
  - 組合流程
  - 呼叫 repository 或第三方服務

- **Repository / DAO**
  - 負責資料庫存取

- **DTO / Schema**
  - 定義輸入輸出格式
  - 避免資料結構混亂

### API 設計原則
1. URL 命名清楚
2. 狀態碼一致
3. 錯誤訊息可讀
4. 權限驗證獨立處理
5. 不把資料庫欄位直接暴露給前端
6. 有版本控制觀念，例如 `/api/v1`

### 實務上我會特別注意
- 不把商業邏輯寫死在 route handler
- 不讓前端直接依賴資料庫欄位命名
- 對外部平台串接做好 timeout、retry、log

這樣之後無論是新增欄位、調整邏輯，或換前端框架，都比較不會互相牽動太大。

---

## 4. 如果網站流量突然增加，哪些地方最容易先出問題？

### 面試官想知道什麼
- 你是否有系統效能觀念
- 你能不能從整體系統角度思考
- 你知不知道瓶頸不一定只在程式

### 建議回答
我會先從幾個常見瓶頸去看：

### 1. 資料庫
最常出問題的是：
- N+1 query
- 沒有 index
- 查詢條件不合理
- 同一頁做太多 join
- 大量寫入造成 lock

### 2. API Server
- 單一 instance 扛不住
- request 同步等待太久
- 某些邏輯做太重
- 串第三方 API 太慢

### 3. 前端
- 首頁載入太多資料
- 圖片沒壓縮
- SSR 頁面查太多資料
- 快取策略沒做好

### 4. 外部服務
像 Amazon API、付款、物流、寄信服務，都可能拖慢整體流程。

### 我的處理方式
我會先做：
- 監控 response time
- 看 error log
- 看 CPU / memory
- 查慢 SQL
- 檢查最近是否有新功能上線

### 常見優化方式
- 加 index
- 分頁
- cache
- background job
- queue
- 圖片 CDN
- 把不必要的同步流程改成非同步

重點不是一開始就講很大的架構，而是先定位瓶頸在哪裡，再對症下藥。

---

## 5. 如何處理第三方平台整合，例如 Google/Facebook/Amazon API？

### 面試官想知道什麼
- 你有沒有串接第三方服務的經驗
- 你是否理解第三方服務不穩定的風險
- 你有沒有保護系統穩定性的意識

### 建議回答
第三方整合我通常會注意這幾件事：

### 1. 把整合邏輯獨立封裝
不要讓 controller 或核心業務邏輯直接依賴第三方 SDK。  
最好做成 service adapter，這樣未來 API 變動比較好調整。

### 2. 做錯誤處理
例如：
- timeout
- retry
- token 過期
- response 格式變動
- 第三方暫時不可用

### 3. 保留 log
要記錄：
- 呼叫時間
- request payload
- response status
- 錯誤訊息
- 對應內部資料 id

### 4. 避免阻塞主流程
如果不是非同步不可，就不要讓使用者一直等。  
例如同步商品資料、批次更新庫存，通常適合用 queue 或 background job。

### 5. 注意權限與安全
像 OAuth 相關整合要注意：
- access token
- refresh token
- callback 驗證
- state 防 CSRF
- 權限範圍控管

我過去有做過 Google / Facebook 登入整合，所以會特別在意 callback 流程、token 驗證與使用者帳號綁定邏輯。

## 6. 如果訂單成立後要同步更新庫存，你會怎麼避免超賣？

### 面試官想知道什麼
- 你是否理解電商核心問題
- 你有沒有併發處理概念
- 你知不知道資料一致性的重要性

### 建議回答
避免超賣是電商系統很核心的問題，不能只靠前端顯示「還有庫存」來判斷。

### 我的做法通常會包含：

#### 1. 下單時以資料庫實際庫存為準
不能信任前端傳來的庫存資訊。

#### 2. 扣庫存要在 transaction 內完成
例如：
- 建立訂單
- 檢查庫存
- 扣減庫存
- 寫入庫存異動紀錄

這幾個步驟最好放在同一個 transaction 中。

#### 3. 使用鎖或原子更新
例如：
- `UPDATE stock SET qty = qty - 1 WHERE sku_id = ? AND qty > 0`

這樣可以避免多人同時搶同一件商品時出現負庫存。

#### 4. 區分保留庫存與實際可售庫存
有些系統會設計：
- available stock
- reserved stock

使用者下單未付款時先保留，超時未付款再釋放。

#### 5. 設計補償機制
如果付款成功但扣庫存失敗，或反過來，也要有 rollback 或人工補償流程。

這題如果答得太理論不夠好，最好講出：
**真正關鍵是資料一致性、交易流程、併發控制與異常處理。**

## 7. 電商網站的搜尋功能，你會怎麼設計？

### 面試官想知道什麼
- 你是否理解搜尋不是單純 SQL LIKE
- 你是否會根據規模選方案
- 你能不能從使用者體驗思考

### 建議回答
如果規模不大，初期可以先用資料庫搜尋，但我會知道這通常只是短期方案。

### 初期可行作法
- 商品名稱 LIKE
- SKU 查詢
- 類別篩選
- 價格區間
- 品牌篩選
- 上下架狀態

### 規模變大後的問題
- LIKE 效能差
- 無法做相關性排序
- 無法支援同義字
- 無法支援拼字容錯
- 無法做複雜篩選

### 中大型系統做法
可導入搜尋引擎，例如：
- Elasticsearch
- OpenSearch

### 可設計的搜尋能力
- 關鍵字搜尋
- 類別 filter
- facet aggregation
- 排序（價格、熱門度、上架時間）
- 關聯推薦
- typo tolerance
- synonym

### 我會怎麼回答得更務實
我不會一開始就說一定要上 Elasticsearch。  
我會先看商品數量、查詢量、營運需求，再決定是否升級搜尋架構。

這樣比較符合實際公司成本與維護考量。

## 8. 如何確保前後端協作順利，不會常常改來改去？

### 面試官想知道什麼
- 你能不能跟團隊合作
- 你有沒有規格管理意識
- 你是不是只站在工程自己的角度

### 建議回答
前後端協作不順，很多時候不是技術問題，而是規格不清楚。

### 我通常會做這幾件事：

1. **先定義 API 規格**
   - request format
   - response format
   - error format
   - 欄位命名規則

2. **使用文件或工具管理**
   - Swagger / OpenAPI
   - Postman collection
   - 簡單的 API spec 文件

3. **先確認資料結構**
   - 哪些欄位前端一定要
   - 哪些欄位只是後端內部使用
   - 哪些欄位未來可能擴充

4. **盡量避免前端直接綁死資料庫結構**
   - 這樣後端調整比較有空間

5. **重要流程先對齊**
   例如：
   - 訂單狀態流轉
   - 退貨流程
   - 庫存更新時機
   - 平台同步規則

我覺得跨部門或前後端合作時，最重要的是把「大家對流程的理解」先講清楚，不然後面會一直返工。

## 9. 如果要你維護別人寫的舊系統，你會怎麼接手？

### 面試官想知道什麼
- 你是否有維護既有系統的耐心
- 你會不會一來就想全部砍掉重寫
- 你是否有風險意識

### 建議回答
接手舊系統時，我不會第一時間就想重寫，因為很多既有系統雖然看起來舊，但通常已經承載實際業務流程。

### 我會先做的事情
1. 先釐清系統用途與主要流程
2. 看目前最常出問題的地方
3. 看 log、程式架構、資料表結構
4. 了解哪些功能是高風險區
5. 先補文件與測試，再逐步調整

### 我的原則
- 優先解決真實痛點
- 避免一次動太大
- 能局部重構就不要一次重寫
- 舊系統與新模組可以漸進共存

我目前工作也有接觸既有 Django 系統重構，所以我會比較傾向用漸進式方式改善，而不是一次性全面替換。

## 10. 如何設計權限系統？

### 面試官想知道什麼
- 你是否理解後台系統常見需求
- 你是否知道權限不是只有登入與否
- 你能不能兼顧安全與維護性

### 建議回答
後台系統的權限設計，我通常會從 RBAC 開始，也就是角色型權限控制。

### 常見結構
- User
- Role
- Permission

### 權限粒度可分成
- 頁面權限
- 功能權限
- API 權限
- 資料權限

### 例如
- 商品管理員：可新增/修改商品，不可刪除
- 訂單人員：可看訂單，不可改商品資料
- 倉儲人員：可更新出貨狀態
- 超級管理員：全部可操作

### 實作時我會注意
1. 前端只做顯示控制，真正授權要在後端
2. 權限判斷不要寫死在各頁面
3. 最好能配置化，方便未來調整角色
4. 保留操作紀錄，方便追查

如果系統未來會越來越大，權限設計越早整理好，後面維護成本越低。

## 11. 如何設計一個穩定的部署流程？

### 面試官想知道什麼
- 你是否有實際上線經驗
- 你是不是只會寫程式，不懂部署
- 你有沒有降低人工作業風險的意識

### 建議回答
我認為部署流程的重點不只是把程式放上去，而是讓上線可重複、可驗證、可回滾。

### 我通常會注意：

#### 1. 環境分離
- dev
- staging / test
- production

#### 2. 設定檔分離
- 不同環境用不同 env
- 密碼與金鑰不要寫死在 repo

#### 3. 基本 CI/CD
至少做到：
- lint
- build
- test
- deploy

#### 4. Docker 化
讓開發與部署環境更一致，降低「我本機可以跑」的問題。

#### 5. 可回滾
如果新版本有問題，要能快速切回上一版。

#### 6. 紀錄與監控
- deploy log
- error monitoring
- service health check

如果公司目前流程還比較手動，我也不會一開始就推很重的方案，而是先從能降低風險的地方開始，例如先把 build、測試與部署步驟標準化。

## 12. 你會怎麼把 AI 用在工程流程裡，而不是只拿來寫程式？

### 面試官想知道什麼
- 你是不是只把 AI 當 buzzword
- 你有沒有實際把 AI 融入工作流程
- 你能不能理性看待 AI 的邊界

### 建議回答
我會把 AI 視為輔助工具，而不是直接取代工程判斷。

### 在工程流程中，我認為 AI 比較適合用在：
1. 協助整理需求與拆解問題
2. 協助閱讀舊程式與理解模組關係
3. 協助產生測試案例
4. 協助初步檢查錯誤訊息與除錯方向
5. 協助撰寫文件、整理 API 說明
6. 協助資料分類或流程判斷的前置處理

### 但我會注意幾件事
- AI 產出不能直接相信
- 關鍵邏輯仍要人工驗證
- 涉及資安或敏感資料要特別小心
- 要先定義清楚任務邊界

我會比較傾向把 AI 放在「提升效率與輔助決策」的位置，而不是讓它直接決定核心商業流程。

# 補充：面試時最容易加分的回答方式

## 1. 不要只講技術名詞
不要一直說：
- 我會 React
- 我會 Node.js
- 我會 Docker

要改成：
- 我用這些技術解決過什麼問題
- 對流程效率或系統穩定性有什麼幫助

---

## 2. 多講「資料流程」與「維護性」
跨境電商公司很在意：
- 商品資料正不正確
- 後台流程順不順
- 系統穩不穩
- 有沒有辦法減少人工操作

---

## 3. 多用這種句型
- 我通常會先釐清流程，再決定技術做法
- 我會優先選擇可維護、風險較低的方案
- 如果是既有系統，我傾向漸進式調整
- 我會把資料驗證與錯誤處理放在前面考慮

---

# 最後整理

這類跨境電商公司的工程師面試，常見技術題不一定很學術，反而偏重：

1. 你是否理解電商流程
2. 你能不能維護既有系統
3. 你會不會做資料匯入、自動化、API 整合
4. 你能不能和主管、營運、設計配合
5. 你回答問題時是否務實、穩定、可信任

真正加分的不是答得多花，而是讓面試官感受到：

> 這個人來了之後，真的能幫我們把系統做穩，並且能理解營運現場需要什麼。
