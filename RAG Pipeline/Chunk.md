# RAG：chunk 怎麼切？PDF/表格怎麼辦？（給非技術讀者的版本）

## 你可以把 RAG 想成「圖書館助理」
- 你的資料（PDF、Word、網頁、表格）就像書
- **chunk** 就像把書拆成「好查的段落卡片」
- 問問題時，系統先把最相關的卡片找出來，再用 AI 把答案整理成「可讀的回覆」

---

## 為什麼要切 chunk？
- AI 一次能讀的字數有限（像一次只能翻幾頁）
- 切得好：更容易**找到正確段落**、回覆更準
- 切得差：段落被切碎或混在一起 → 找不到、答錯、或引用亂跳

---

## chunk 切法的三個原則（最重要）
### 1) 以「一個意思」為單位（semantic chunk）
- 不要死切 500 字、1000 字
- 以「一個概念/一段流程/一條規則」為一塊
- 讀起來像一個完整小段落：有主題、有結論

### 2) 跟著章節走（section-aware）
- 文件本來就有章節標題：就用它當邊界
- chunk 內保留「我在哪一章、哪一節」的資訊（metadata）
  - 例如：文件名 / 章節標題 / 小節標題 / 頁碼

### 3) 少量重疊（overlap）避免斷章取義
- chunk 之間留一點重疊文字（例如 50~150 字）
- 用來避免「關鍵句剛好在邊界被切掉」
- 重疊太多會浪費成本、也可能讓檢索結果重複

---

## 不同文件型態：不同 chunk 策略（你可以用這個決策樹）
### A) 長篇文章/規格文件（Word、Confluence、網頁）
- 優先：**section-aware + semantic chunk**
- chunk 大小建議（大概概念）
  - 每塊 200~600 字（或 1~3 段落）
  - 遇到「條列規則」可更短（例如每條規則 1 chunk）

### B) SOP/流程文件（步驟很清楚的）
- 以「一段流程」為 chunk
- 每個 chunk 包含
  - 流程目的
  - 主要步驟
  - 例外/注意事項（若有）

### C) FAQ/問答型文件
- 以「一組 Q&A」為 chunk
- metadata 放上
  - 問題類別 / 關鍵字標籤

### D) 程式碼/技術文件（README、API 文件）
- 以「一個功能區塊」切
  - 例如：一個 endpoint、一個 config、一段 usage
- chunk 內保留 code block（但不要把整個 repo 都塞進單一 chunk）

## PDF 最常見的坑：排版不是內容 😅
### PDF 的問題
- PDF 是「視覺排版」格式，不是「語意結構」格式
- 常見災難：
  - 欄位（雙欄排版）被讀成一行亂序
  - 頁眉頁腳、頁碼被混入正文
  - 表格被拆成一堆碎字

### PDF 的處理策略（建議流程）
1) **Parse & Normalize**
   - 先把頁眉頁腳、頁碼、重複標題去掉
2) **保留頁碼與段落位置**
   - 方便後面做 citations（引用來源）
3) **section-aware 優先**
   - 盡量用 PDF 的「標題階層」或你自己偵測的標題樣式（字體大小/粗體/編號）
4) **必要時：用「段落重建」**
   - 把被斷行的句子接回去（例如每行換行但其實是一句話）

## 表格怎麼辦？重點是「保留欄位語境」✅
### 為什麼表格不能直接當文字切？
- 表格的意思常常靠「欄名」決定
- 如果你只留下數值或某一格文字，會失去語境
  - 例如：`10` 是價格？數量？天數？罰鍰？完全不知道

### 正確做法：table extraction + 兩種呈現方式（通常一起做效果最好）
#### 方式 1：表格「轉成可讀的列描述」（推薦給問答）
- 把每一列變成一段文字，並把欄名一起帶上
- 範例（概念）
  - 原表：| 產品 | 價格 | 幣別 |
  - 轉文：`產品=ABC；價格=10；幣別=TWD`

優點
- 檢索更容易命中（embedding 對自然語言通常比較友善）
- 回答時容易引用

#### 方式 2：保留原表格（Markdown/CSV）作為附件內容
- chunk 內可以放「表格本體」（Markdown table 或 CSV 片段）
- metadata 記錄
  - 表格標題、所在頁碼、欄位清單

優點
- 需要原始對照時不會失真（例如要對欄位、對數值）

### 表格 chunk 的切法建議
- 不要把「整張大表」塞成一個 chunk（太大）
- 用「行群組」切：
  - 每 chunk 包含：欄名 + 20~50 列（視大小）
- 若表格有分區（例如依年份/地區分組），就依分區切（section-aware）

---

## semantic chunk / section-aware / overlap 三者怎麼搭配？
- **最常用的穩健組合**：
  - 先 section-aware（用標題定大邊界）
  - 再 semantic chunk（在章節內按語意切小塊）
  - 最後 overlap（在邊界加少量重疊）
- **不要**只靠固定字數切（容易把意思切斷）
- **不要** overlap 太大（會變貴、也會出現重複命中）

---

## chunk 之外，還有兩個「容易被忽略但很關鍵」的設計
### 1) metadata（文件身分證）
每個 chunk 建議至少包含：
- doc_id / 文件名
- 標題階層：H1/H2/H3（或章/節）
- 頁碼（PDF）
- 表格：table_id、欄位清單（若是表格 chunk）
- 更新時間 / 版本（方便回滾）

### 2) citations（答案的「出處」）
- 回答時要能說清楚：這句話來自哪個文件、哪一頁、哪一節
- 對企業內部使用尤其重要（可追溯、可稽核）

## Next.js + NestJS 在 RAG 系統裡可能扮演的角色（非技術版）
### Next.js：使用者看到的「前台」
- 聊天介面（像企業版 ChatGPT）
- 文件搜尋/瀏覽/預覽（點一下就能看引用來源）
- 權限控制的 UI（不同人看到不同資料）
- 管理頁面（上傳文件、查看索引狀態、版本）

### NestJS：系統背後的「總控台與管家」
- API 入口：接收問題、回傳答案
- 身分驗證/授權（SSO、JWT、RBAC）
- 檢索編排（呼叫向量庫、混合檢索、rerank）
- Prompt 組裝（把相關 chunk 與規則整理成「可控的輸入」）
- 日誌/監控/稽核
  - 問了什麼、用到哪些 chunk、回覆了什麼（可追蹤）
- 佇列與背景工作（Ingestion）
  - 文件解析、切 chunk、embedding、寫入索引（可用 queue 做非同步處理）

### 一句話總結
- **Next.js：把 AI 能力做成好用的產品介面**
- **NestJS：把資料、權限、檢索、流程控管成可靠的後端服務**

## 實務落地的小抄（你可以直接照做）
### 最小可行版本（MVP）
- 文件：先從「結構清楚的 Word/網頁」開始
- chunk：section-aware + semantic chunk + 小 overlap
- 表格：先做「列描述」版本
- citations：先做到「文件名 + 頁碼/章節」
- 前台：Next.js 聊天頁 + 引用來源抽屜
- 後台：NestJS 做 API + ingestion 任務

### 進階（品質/可靠性）
- PDF 做頁眉頁腳清理 + 段落重建
- 表格同時保留「列描述」+「表格本體」
- 多索引：向量 + 關鍵字（混合檢索）
- 版本化：每次索引產出 manifest，可回滾

---

## 常見錯誤（避雷）
- 把整份 PDF 當一個 chunk（幾乎一定檢索不到）
- 表格只存數值、不存欄名（會失去欄位語境）
- chunk 沒 metadata（答案無法引用、也難除錯）
- overlap 開太大（成本飆升 + 內容重複命中）
